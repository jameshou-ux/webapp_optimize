# imToken Web · 首页改版纲要
**日期：** 2026-04-30  
**产品：** web.token.im  
**状态：** Beta · Arbitrum 主网  
**阶段：** 第二轮审计 + 设计方向

---

## 1. 产品定位

> 就像 Kindle 之于书籍——人们用 Kindle 随时随地阅读和享受电子书。
> imToken Web App 的目标是：让用户随时随地**阅读和享受 Token**（数字货币及更多）。

**是什么：** Token 的网页门户 · 轻量级应用 · Token 与区块链的探索型界面  
**不是什么：** 又一个加密钱包 · 静态网站 · 密钥管理工具 · 需要预先安装的应用

**Just-in-time 访问：浏览器 + 网络连接 = 从任何地方访问**

**愿景：**
> "从孩童到老人，只要能上网浏览，你就始终拥有平等进入代币化世界的机会。"

---

## 2. 核心能力与价值主张

| 能力 | 对用户意味着什么 |
|---|---|
| 浏览器原生访问 | 无需安装 App 或插件，任意现代浏览器 + 网络即可全功能使用 |
| Passkey 账户 | 用指纹/面容 10 秒创建钱包，无助记词焦虑 |
| Send by Link | 生成可领取链接；接收方无需提前有钱包，点击即可创建账户并领取 |
| Gas 抽象 / Paymaster | 用 USDT 等稳定币支付 Gas，无需先购买 ETH |
| Token Collection 模型 | 按主题策展 Token——通过 Collection 探索 Token 宇宙，而非面对原始链地址 |

### Send by Link——最核心的差异化能力

**它解决了什么问题：**
- 传统世界：无法向没有账户的人转账；微信用户无法向 PayPal 用户转账
- 加密世界：无法向没有钱包的人发 Token；无法通过 Web2 渠道发送；无法在没有原生 Gas Token 的情况下领取

**解决方案：** 发送方生成链接 → 通过任意渠道发送（消息、邮件、WhatsApp、QR 码）→ 接收方点击链接，即时创建账户并领取 → **任何人都可以向任何人发送 Token，无需中介授权**

**典型场景：**
- **项目方**——通过领取链接向新用户分发 Token，引导新持有者
- **发送方**——给刚入圈的朋友发 Token，无需对方提前建钱包
- **邀请方**——通过 WhatsApp 发可领取链接，邀请朋友体验 Token 功能

## 3. 目标用户

| 用户类型 | 特征 | 首页需要提供什么 |
|---|---|---|
| **加密新手** | 不熟悉钱包、公链、Gas；习惯 Web2 登录方式（Google、Apple、微信） | Passkey 等熟悉的登录方式、引导式入门、情境化教育内容，不用行话 |
| **加密用户** | 持有 Token、关注价格、阅读市场信息，希望了解 Token 用途 | Token Collection、市场背景、持仓盈亏、Token 实用功能、个性化资讯、发送和邀请新用户 |

次要用户：
- **Token 分发方 / 项目方**——需要轻量的代币分发和活动 Onboarding 工具
- **已有 imToken App 用户**——可能信任品牌，但需要理由使用 Web 版

## 4. 为什么首页转化现在尤其重要

当前用户量有限，首页是整个产品中杠杆最高的转化界面。

**当前漏斗（实测）：**
访客进入 → 看到漂浮气泡，无价值主张 → 找不到开始入口 → 活动流量无着陆路径 → 大多数访客无收益地离开

**关键指标（当前状态）：**

| 指标 | 现状 |
|---|---|
| 主 CTA（填充按钮） | 0 个 |
| 首屏可见价值主张 | 0 条 |
| 可滚动内容 / Footer | 无 |
| 感知首屏加载时间 | 约 3 秒 |

> 首页是为已经理解产品的回访用户设计的——但这类用户是少数。每一个新访客都是被浪费的机会。

## 5. 首页的三大核心职责

这三个职责既是首页改版的目标，也是当前最主要的缺陷所在。

### 职责一：产品价值传递
让匿名访客在 5 秒内理解「imToken Web 是什么、能做什么、为什么值得一试」。

- 用行为承诺替换抽象口号
- 把 Passkey、Send by Link、Gas 抽象等差异化能力显性化
- 建立信任信号（生态、安全、Roadmap）

### 职责二：用户 Onboarding
为不同入场路径的用户（主动访问 / 活动链接 / 朋友邀请）提供清晰的账户创建动机和路径，并在注册后立刻完成第一个激活动作。

- 主 CTA 首屏可见
- 活动和链接流量有专属承接入口
- 注册成功后立刻给出首个任务，而不是空白页

### 职责三：功能引导与使用
让用户在探索过程中自然发现和使用产品功能，而不是面对「看不懂的气泡」或空白状态。

- Token Collection 附带「为什么重要」的上下文
- Spatial View 升级为可行动的探索界面
- 回访用户看到与自身资产和行为相关的内容

## 6. 当前首页：三大职责的缺口

### 页面结构（Desktop 实测）
- **导航：** Logo · 搜索 · My Holdings · My Collections · 账户图标（右上角）
- **Hero：**「Home of Tokens — Digital assets under control」
- **主视觉：** 漂浮气泡（Collection 分类名称），中央是「My Holdings」气泡
- **首次弹窗：** 介绍 Passkey + Send by Link（可跳过）
- **首屏以下：** 空白。无 Footer，无滚动内容，无信任信号，无 FAQ

## 7. 改进方向

两个相对独立的设计问题，各有两套方案选项。

---

### 大方向 A：Onboarding 与价值传递

**核心问题：** 匿名用户进入首页后，看不懂产品、找不到注册理由、没有主 CTA。

#### 方案 A1：逐屏滚动叙事（参考 Prototype v3 Direction 3）

**设计思路：** 用户第一次访问时，不一次性看到所有产品卖点，而是顺着 scroll 一次只接收一个 key point。每一屏左侧只出现一个核心句子和一段解释，右侧 sticky 视觉舞台保持在同一位置，通过一致的 morph / fade / path 动画从一个状态平滑过渡到下一个状态。用户连续向下滚动到最后一屏时，视觉舞台无缝扩展为正式 Spatial View 主界面。

核心体验不是「长页面介绍」，而是「一步一步进入产品」：

> Entry → Send by Link → Access anywhere → Passkey / account → Collections → Spatial View

交互结构：
- 页面采用 `44% copy rail + 56% sticky visual stage` 的左右结构。
- 左侧每个 scene 高度接近 100vh，一次只展示一个 key point，不同时展示多个章节。
- 右侧 visual stage 全程 sticky，用户滚动时只切换当前 active scene 的视觉状态。
- 每个 scene 的标题、正文、CTA 使用一致的 reveal 动画：轻微上移 + fade in。
- 视觉状态使用统一动效语言：同一个中心 Token / imToken orb 逐步变形、连线、扩散、聚合，避免每屏像完全不同的插画。
- 最后一屏左侧 copy rail 逐渐退场，右侧视觉区扩展为完整 Explore / Spatial View 首页。

章节内容建议：
1. **Entry**：「A lighter way to start onchain」— 先降低进入门槛，不急着解释所有能力。
2. **Send to anyone**：Send by Link，发 Token 像发链接，接收方无需提前有钱包。
3. **Access anywhere**：浏览器 + 网络连接即可访问，不依赖特定设备或预安装。
4. **Create with Passkey**：用设备解锁创建账户，无助记词作为第一步。
5. **Start with collections**：从 Token collections 开始理解 Token，而不是先面对链和合约地址。
6. **Land in Spatial View**：最后自然进入正式 Explore 主界面，出现主 CTA「Create with passkey」和「Explore tokens」。

滚动节奏：
- One scroll = one key point。
- 每一屏的信息量必须足够少，用户能在 3-5 秒内读完。
- 每屏只保留一个主动作或弱 CTA，最终 CTA 在最后一屏集中强化。
- 不应把 4-6 个章节同时铺在同一张图里；草图应表现「当前屏 + 下一屏隐约延续」或「分屏状态序列」。

**适用场景：** 首次访问用户；希望最大化价值传递深度；品牌叙事优先。

**权衡：** 沉浸感强，能把用户从概念带入产品；但需要用户愿意滚动，且回访用户必须有跳过机制，例如 header 中常驻「Skip to Explore」或记住已完成 onboarding。

---

#### 方案 A2：Modal 滑动传递价值主张

**设计思路：** 用户进入首页后，一个全屏或大尺寸 Modal 以卡片轮播方式呈现产品核心价值，最后一屏给出「创建账户」或「跳过直接探索」两个出口。

交互结构：
- Modal 覆盖首页主内容（可见背景虚化的 Spatial View）
- 3–4 张卡片，每张聚焦一个能力点，支持横向滑动或点击翻页
- 最后一张：主 CTA「创建账户」+ 次级「跳过，先探索」
- 再次访问时不再自动弹出（localStorage 标记）

卡片内容建议：
1. 「浏览器里的 Token 门户」— 产品一句话定位
2. Passkey 账户 — 无助记词，10 秒创建
3. Send by Link — 发给任何人，接收方无需提前有钱包
4. 「创建账户」或「先逛逛」

**适用场景：** 不希望大幅改动主界面结构；给回访用户快速跳过的灵活性；工程改动量较小。

**权衡：** 实现成本低，但 Modal 有被跳过的风险，信息密度受限。

---

**两个方案的共同要素（无论选哪个都需要）：**
- 「收到链接了？领取你的 Token」常驻入口，承接活动和朋友分享流量
- Send by Link 在 Onboarding 尾部作为增长钩子：每个激活用户都生成一条链接 → 接收方复制 Onboarding 路径 → K-factor > 1

---

### 大方向 B：首页视图改版

**核心 IA 共识（两个选项共同的前提）：** 将 **Explore** 和 **My Holdings** 作为首页最顶层的两个切换 Tab，明确区分「探索发现」与「我的持仓」两种使用意图。这是 v3 Direction 2 的核心 IA 判断，与选项 B1 / B2 无关，应无条件采纳。

```
导航栏：[Logo] [搜索]  [Explore | My Holdings]  [Spatial/List 切换]  [登录] [创建账户]
                         ↑ 顶层主 Tab，明确两种意图
```

---

#### 方案 B1：Spatial + List 统一 Shell，信息增强（参考 Prototype v3 Direction 2）

**设计思路：** 保留 Spatial View 作为品牌差异化资产，但将 Spatial 和 List 整合进同一个 Shell，共享顶层导航和 Companion Panel。两种视图切换时内容状态保持连续，不是两套分离体验。

核心改进：
- **Spatial View**：气泡附带真实价格 + 24h 涨跌幅；气泡大小映射市值/相关度；选中 Collection 后右侧展开 Companion Panel（Collection 背景、相关 Token、行动 CTA）
- **List View**：Collection 卡片增加「为什么重要」一句话理由、Top Mover、安全/状态标签、上次更新时间
- **Companion Panel**（两种视图共享）：Campaign 详情、Collection 摘要、Trust cue、主 CTA
- **Campaign Hub**：Spatial 中心区或 List 顶部 Banner 位可配置为活动入口

**适用场景：** 希望保留 Spatial View 的品牌识别度；Spatial 有忠实用户群；长期想做空间探索的深度功能。

**权衡：** 工程和设计复杂度更高；Spatial 的学习曲线对新用户仍然存在。

---

#### 方案 B2：取消 Spatial，回归结构化列表视图

**设计思路：** 放弃 Spatial View，以一套清晰的结构化列表页作为唯一主视图。首页即是功能界面，无需在「好看」和「好用」之间平衡。

核心改进：
- Explore Tab：Collection 卡片列表，附带完整信息层（价格、Insight、理由、CTA）
- My Holdings Tab：持仓概览、活动记录、待处理链接
- Campaign 模块：List 顶部可配置 Banner + Collection 内嵌 Campaign 卡片
- 搜索和筛选功能前置，支持按类别、链、风险等级过滤

**适用场景：** 优先考虑清晰度和可操作性；移动端体验更重要；希望降低工程维护两套视图的成本。

**权衡：** 放弃了 Spatial 的差异化，产品辨识度可能下降；现有用户可能有流失风险。

## 总结

> 当前首页在对已经相信产品的人说话——把 99% 的新访客当成了已经了解产品的用户。
>
> 改版的核心不是视觉升级，而是让首页同时完成三件事：**说清楚产品价值、引导用户完成注册、让功能在探索中自然可见。** 三件事做到位，Home of Tokens 才真正名副其实。
